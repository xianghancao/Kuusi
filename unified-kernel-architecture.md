# Lumen 统一内核 + 多视图编辑器 — 产品与架构说明书

> 本文档是 Lumen 的唯一说明书和架构基准。任何新功能开发前，先判断它是否落在统一内核内：落在内核内，可以做；需要另起独立系统，应拒绝或作为外部工具对接。

---

## 1. 一句话定义

Lumen 是一个以结构化知识树为内核的即时渲染编辑器。它先以思维导图视图呈现：每个节点都是可编辑、可渲染的富内容块；未来同一份数据还能投影为文档视图和大纲视图。

核心不是“画布工具”，而是：**一份 block 树，多种视图，同一套公式、图片、代码、表格和 AI 结构操作能力。**

---

## 2. 不可违背的核心原则

1. **单一事实来源**：所有内容最终只存一份结构化 block 树。视图不是独立编辑器，而是这棵树的不同投影。
2. **块类型全局通用**：公式、图片、代码、表格、任务等内容类型只在内核注册一次，所有视图共享同一套渲染和编辑逻辑。
3. **即时渲染，原地编辑**：没有源码/预览双栏。用户在内容所在位置编辑，也在同一位置看到最终效果。
4. **键盘优先**：思维导图视图的核心路径必须可以不碰鼠标完成：输入、`Tab` 生子节点、`Enter` 生兄弟节点、自动布局、光标跳转。
5. **AI 操作结构，而非裸文本**：AI 看到的是块的父/子/兄弟和根路径，返回的是可 hydrate 的结构化 JSON，而不是一段散文。

---

## 3. 核心用户体验

### 3.1 思维导图输入流

打开软件时，画布中央只有一个根节点，光标已经在节点里。用户直接打字，这就是渲染态，没有独立预览窗口。

- `Tab`：在右侧自动生成子节点，建立连线，dagre 自动布局，光标跳进新节点。
- `Enter`：在同一层生成兄弟节点，自动布局，光标跳进新节点。
- `Shift + Enter`：在当前节点内换行。
- 单击节点：选中。
- 双击节点：编辑。
- 拖入/粘贴图片：在节点内直接显示，可调整大小。
- 折叠节点：隐藏后代节点和连线，但保留结构。

这个流程是 v1 的第一优先级。工具栏和鼠标操作只是辅助入口，不能破坏键盘主路径。

### 3.2 公式体验

公式体验是 Lumen 区别于普通思维导图和 Markdown 工具的关键。

- 输入 `$LF_t = \frac{Lev_t}{Lev_{t-1}}$` 时，公式源码应在节点内原地可见、可编辑。
- 光标移出公式范围后，公式应在原位置渲染成 KaTeX 排版效果。
- 光标点回公式时，应在原位置恢复源码，继续编辑 `\frac{...}{...}`。
- 单独一段 `$$x=1$$` 应渲染为块级公式；行内 `$x=1$` 应保持行内公式。
- `text $$x=1$$` 不应被误识别为行内公式。

最终目标是“即时渲染 + 原地编辑”的 inline math/editable math node，而不是双栏预览，也不是把节点变成完整文档编辑器。

### 3.3 富内容保真

节点内容必须保留结构，不得降级为纯文本：

- 富文本：粗体、斜体、列表、链接等基础能力。
- 公式：行内 `$...$` 与块级 `$$...$$`。
- 图片：粘贴、拖入、base64 单文件保存，支持等比例缩放。
- 代码：原地语法高亮，不执行。
- 后续：表格、任务列表、行内高亮、Markdown 快捷输入。

## 4. 统一内核

### 4.1 目标数据模型

长期目标是整份文档就是一棵 block 树：

```ts
type Block = {
  id: string;
  type: BlockType;
  content: ProseMirrorJSON;
  children: string[];
  parent: string | null;
  attrs?: Record<string, unknown>;
};
```

- `children` / `parent` 表示知识结构。
- `content` 保存结构化富内容，不保存 Markdown 字符串。
- `attrs` 保存块级语义状态，不保存画布坐标、节点尺寸、折叠等视图状态。
- edges 不持久化为事实来源，打开文件时由 block 树生成。

### 4.2 当前 v1 过渡模型

当前实现允许使用过渡模型，但必须逐步向“知识结构”和“视图状态”分离迁移。节点数据只保存知识结构，不应长期保存画布坐标。

```ts
type MindNode = {
  id: string;
  content: ProseMirrorJSON;
  children: string[];
  parent: string | null;
};
```

画布坐标、节点尺寸、折叠状态、布局间距等属于思维导图视图状态，存在独立的 `viewState`：

```ts
type SheetViewState = {
  sheetId: string;
  layoutSpacing: "compact" | "standard" | "spacious";
  nodes: Record<
    string,
    {
      position?: { x: number; y: number };
      dimensions?: { width: number; height: number };
      collapsed?: boolean;
    }
  >;
};
```

这样做的原因：文档视图不需要 x/y，AI 生成子树也不应编造画布坐标。坐标缺失时由自动布局重新计算即可。当前代码仍把部分视图状态放在 `MindNode` 上，下一阶段应优先迁移到 `viewState`。

这个模型已经具备 block 树的核心形状，但还不是完整统一内核。v2 前应迁移到正式 `Block` schema，使文档视图、思维导图视图和 AI 层共享同一套数据。

### 4.3 块类型注册

| 块类型 | 渲染 | 编辑 | 阶段 |
|--------|------|------|------|
| `text` | 富文本、行内公式、行内高亮 | 即时渲染，点入源码 | v1 |
| `equation` | 块级公式，KaTeX | 原地编辑源码 | v1 |
| `image` | 粘贴/拖入即显示 | 原地缩放 | v1 |
| `code` | 语法高亮 | 原地编辑，不执行 | v1 |
| `table` | 表格 | 原地编辑 | v2 |
| `task` | 可勾选任务项 | 原地编辑 | v2 |
| `reference` | 外部资源引用 | 只读或跳转 | v4 |

新增块类型只能在内核注册一次，不能为某个视图单独复制一套。

### 4.4 文件格式与迁移

保存格式为单文件 JSON，扩展名可继续使用 `.mindmap.json`，后续可升级为更通用的 `.lumen.json`。

多 sheet 语义必须明确：一个文件可包含多个 sheet，每个 sheet 是一棵独立 block 树，不是同一棵树的多个投影。多个 sheet 之间不共享节点。

目标文件格式从 `1.1` 开始拆分 `sheets` 和 `viewState`：

```json
{
  "version": "1.1",
  "active_sheet_id": "sheet-1",
  "sheets": [
    {
      "id": "sheet-1",
      "title": "Sheet 1",
      "root_id": "node-1",
      "nodes": {}
    }
  ],
  "viewState": {
    "sheet-1": {
      "layoutSpacing": "standard",
      "nodes": {}
    }
  }
}
```

- `sheets[].nodes` 只存知识结构。
- `viewState[sheetId].nodes` 存该 sheet 的画布坐标、尺寸和折叠状态。
- `viewState[sheetId].layoutSpacing` 存该 sheet 的布局间距。

原则：

- 所有富内容保存为 ProseMirror/Tiptap JSON。
- 图片 v1 可用 base64 data URL，保证单文件打开最可靠。
- `version` 必须保留，用于 schema migration。
- 旧版 `1.0` 文件可以迁移读入；读入时把节点上的坐标、尺寸、折叠状态迁移到 `viewState`。
- 未知块类型不能直接丢失，应以 fallback block 保留原始 JSON。

## 5. 视图投影

视图只负责“如何呈现和编辑 block 树”，不拥有独立数据源。

| 视图 | 投影方式 | 适合 | 阶段 |
|------|----------|------|------|
| 思维导图视图 | block 树 -> 画布节点 + 连线 + 自动布局 | 结构梳理、概览、快速生长树 | v1 |
| 文档视图 | block 树 -> 线性文档 | 长文写作、表格、推导、导出 | v2 |
| 大纲视图 | block 树 -> 折叠缩进列表 | 快速重排层级 | v4 |

### 5.1 视图契约

每个视图必须遵守：

- 只能通过内核 API 修改 block 树。
- selection/focus 可以是视图状态，但必须能映射回 block id。
- 折叠、布局距离、画布位置、节点尺寸默认属于 `viewState`；只有真正改变知识结构语义的状态才进入 block attrs。
- 任一视图修改内容后，其他视图应能从同一棵树实时投影出来。

### 5.2 思维导图视图要求

- 使用 React Flow 呈现节点和连线。
- 使用 dagre 进行自动树布局。
- 根据实际节点尺寸参与布局，避免节点遮挡。
- 支持三档布局间距：紧凑、标准、宽松。
- 支持最多 20 个 sheet。
- XMind 导入已经支持，可继续保留；但它不是 v1 核心体验的阻塞项，导入结果必须转换为内核 block 树。

## 6. 编辑状态与焦点管理

Tiptap/ProseMirror 和 React Flow 会争抢键盘事件，这是 v1 最容易出问题的地方。

状态规则：

- 任一时刻最多只有一个节点处于编辑态。
- 单击节点只选中，不编辑。
- 双击节点进入编辑态。
- 打开、新建、切换 sheet 后，root 自动进入编辑态。
- 编辑器内 `Tab` 创建子节点。
- 编辑器内 `Enter` 创建兄弟节点。
- 编辑器内 `Shift + Enter` 才换行。
- 中文输入法组词期间，不得把 `Enter` 误判为创建节点。
- 编辑态下删除键只删除内容，不删除节点。
- 浏览态下 `Delete`/`Backspace` 删除选中节点，但 root 永远不可删除。

焦点冲突必须优先验证，不能靠偶然行为。

## 7. AI 贯穿层

AI 不是某个视图的按钮，而是作用于内核 block 树的一组能力。

### 7.1 结构化上下文

每次 AI 操作前，构造当前块上下文：

- 从 root 到当前块的路径，展平成 outline。
- 当前块的父块。
- 当前块的兄弟块。
- 当前块已有子块。
- 当前块内容和块类型。

模型因此知道当前内容在整棵知识树中的位置。

### 7.2 AI 能力

1. **补全子树**：选中一个块，AI 基于结构位置生成若干子块，返回 JSON 子树，hydrate 进内核。
2. **续写/展开**：在文档视图中基于上文结构续写当前块。
3. **选中改写**：选中任意块或文本，AI 润色、翻译、精简，保留块类型与结构位置。

### 7.3 调用约定

- 结构补全要求模型返回 JSON 块树，不返回散文。
- 返回内容必须经过 schema 校验再写入内核。
- 支持流式接收，边生成边 hydrate 或占位渲染。

## 8. 外部边界

以下能力永远不进入 Lumen 内核：

| 需求 | 归属 | 对接方式 |
|------|------|----------|
| 代码执行 | JupyterLab / 外部运行时 | Lumen 只展示 code block，不执行 |
| 完整 LaTeX 编译 / PDF 排版 | MyST CLI / Overleaf | 导出 Markdown/MyST 后交给外部工具 |
| Jupyter 产物 | 外部输入 | 通过 Markdown/MyST 或只读引用进入 |
| 云同步/多人协作 | 独立服务 | 不属于本地 v1/v2 内核 |

判断标准：需要另起运行时、编译器、协作服务的，一律在边界外。

## 10. 演进路线

每一步停下时，都必须是可独立使用的完整工具，不接受半成品。

### v1：思维导图视图

目标：成为可持续使用的本地结构化思维导图工具。

必须完成：

- 键盘优先输入流：启动编辑 root，`Tab` 子节点，`Enter` 兄弟节点。
- 节点内即时渲染：公式、图片、代码。
- 单击选中，双击编辑。
- 自动布局、布局间距菜单、节点尺寸参与布局。
- 节点折叠。
- 多 sheet，最多 20 个。
- 保存/打开 `.mindmap.json`。
- XMind 导入已支持，但不是 v1 核心验收阻塞项。
- 将坐标、尺寸、折叠状态迁移到 `viewState`，避免污染知识结构。

核心验收：

- 50-100 个节点仍可流畅编辑和自动布局。
- 保存后重新打开，内容、图片、公式、代码、层级、折叠状态都保真。
- 编辑态下键盘不会误删或移动节点。
- root 不能被删除。

### v2：统一 block 内核 + 文档视图

目标：从 `MindNode` 过渡模型迁移到正式 block schema，并增加文档视图。

必须完成：

- `MindNode` -> `Block` migration。
- 文档视图与思维导图视图共享同一份 block 树。
- 表格、任务列表进入内核。
- 公式即时渲染体验达到“点入源码、移出渲染”。
- 文件格式升级与旧文件迁移。
- XMind 导入继续转换为内核 block 树，并适配新 schema。

### v3：AI 贯穿层 + MyST

目标：让 AI 作用于结构，而不是作用于一段孤立文本。

必须完成：

- 结构化上下文构造。
- AI 补全子树。
- AI 改写选中块。
- MyST / Markdown 导入导出。
- schema 校验和安全 hydrate。

### v4：大纲视图与外部引用

目标：完善多视图工作流，并接通科研写作生态。

可选内容：

- 大纲视图。
- Jupyter notebook 只读引用。
- 外部资源 block。
- 自定义块类型。

## 11. v1 验收清单

最小测试场景：

1. 打开软件，root 位于画布中央并自动进入编辑。
2. 直接输入文字，按 `Tab` 连续创建子节点，按 `Enter` 创建兄弟节点。
3. 每次创建后自动连线、自动布局、光标跳进新节点。
4. 输入 `$E=mc^2$` 和 `$$x=1$$`，行内/块级公式区分正确。
5. 拖入图片，图片进入节点而不是浏览器新开页面。
6. 图片可等比例缩放。
7. 输入代码块，代码原地高亮。
8. 折叠节点后，后代节点和连线隐藏；展开后恢复。
9. 新建、保存、打开后内容和结构完全保真。
10. 删除选中节点时，root 不可删除。
11. 编辑态下 `Delete` 只删除内容，不删除节点。
12. `Cmd/Ctrl + Z` 可撤销节点操作；编辑器内部保留原生文本撤销。
13. 保存后的节点数据不含画布坐标、尺寸和折叠状态；这些信息存在 `viewState`。删除某节点的 `viewState` 后重新打开，自动布局能重新算出位置且内容不丢。

## 12. 每次加功能前必答

> 这个功能，是一种块、一种视图投影、一种内核能力，还是需要另起独立系统？

- 是块：注册到统一内核，所有视图共享。
- 是视图：只做 block 树投影，不持有独立数据。
- 是 AI 能力：输入/输出必须是结构化上下文和 JSON。
- 是外部系统：不要塞进内核，通过导入、导出、引用或外部工具对接。
